# Auth機能 結合試験項目書 (結合 Test Plan)

## 1. 試験実施方針

- **目的**: ユーザー登録からログイン、および保護されたリソースへのアクセスまでの一連の「認証の鎖」が、フレームワークの機能（Validation, Guard, Strategy）および実DB（PostgreSQL）を含めて正しく機能することを担保する。
- **範囲**: `AuthController`, `AuthService`, `UserService`, `JwtStrategy`, `JwtAuthGuard`
- **手法**: Vitest + Supertest + 実DB (Prisma)

---

## 2. 結合試験項目 (正常系)

| カテゴリ         | テストケース             | 確認事項 (Assertion / Expectations)                                                                                                                                           | ステータス |
| :--------------- | :----------------------- | :---------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :--------- |
| **登録**         | 正しいデータでの新規登録 | ・DBにUserレコードが作成されること<br>・レスポンスに`accessToken`が含まれること<br>・**Cookieに`refresh_token`が設定され、`httpOnly: true`, `sameSite: 'lax'`が含まれること** | `201`      |
| **ログイン**     | 正しいデータでの認証     | ・有効なアクセストークンが返却されること<br>・レスポンスに`refreshToken`が含まれないこと<br>・**Cookieに`refresh_token`が適切に設定（更新）されること**                       | `200`      |
| **リフレッシュ** | 正常なトークン更新       | ・Cookieの`refresh_token`を用いて新しい`accessToken`が取得できること<br>・新しい`refresh_token`がCookieに上書きされること                                                     | `200`      |

## 3. 結合試験項目 (異常系)

| カテゴリ                     | テストケース         | 確認事項 (Assertion / Expectations)                                                           | ステータス |
| :--------------------------- | :------------------- | :-------------------------------------------------------------------------------------------- | :--------- |
| **登録: バリデーション**     | 不正な形式での入力   | ・`username`が空、または`password`が短すぎる場合にバリデーションエラーが発生すること          | `400`      |
| **登録: 重複チェック**       | 既存usernameでの登録 | ・既に存在する`username`を指定した場合、`UsernameAlreadyExistException`によりエラーとなること | `409`      |
| **登録: 重複チェック**       | 既存emailでの登録    | ・既に存在する`email`を指定した場合、`UserEmailAlreadyExistException`によりエラーとなること   | `409`      |
| **ログイン: バリデーション** | 必須項目の欠落       | ・リクエストボディが不完全な場合、認証処理の前に400エラーが返ること                           | `400`      |
| **ログイン: 認証失敗**       | 誤った認証情報       | ・存在するユーザーに対し誤ったパスワードを入力した場合、JWTが発行されないこと                 | `401`      |
| **リフレッシュ: 認証失敗**   | Cookieなし           | ・`refresh_token`が存在しない場合、認証エラーとなること                                       | `401`      |
| **リフレッシュ: 認証失敗**   | 不正トークン         | ・不正または改ざんされた`refresh_token`の場合、認証エラーとなること                           | `401`      |

## 4. Guard / Strategy 試験項目 (認可)

| カテゴリ         | テストケース       | 確認事項 (Assertion / Expectations)                                                            | ステータス |
| :--------------- | :----------------- | :--------------------------------------------------------------------------------------------- | :--------- |
| **JwtAuthGuard** | 有効なトークンあり | ・`Authorization: Bearer <valid_token>`を付与して保護APIを叩いた際、正常にレスポンスが返ること | `200`      |
| **JwtAuthGuard** | トークンなし       | ・ヘッダーにトークンを含めない場合、アクセスが拒否されること                                   | `401`      |
| **JwtAuthGuard** | 不正なトークン     | ・署名が改ざんされたトークンや期限切れトークンの場合、アクセスが拒否されること                 | `401`      |
