認証試験観点
user機能とauth機能が存在するが、userにはget、createしかなくドメインロジックが薄いため単体試験は
おこなわない。
authには、hash化とcompareがあることと、重複チェックがあるのでその試験を簡単にすること。

次に結合だが、ロジックが薄くmockをなくしたときに問題が起こることがないと判断し、結合は行わない。
よってE2Eをメインにして少し単体を触るくらいの試験をする。

単体（authのみ）

登録
 バリデーション
  name
   失敗：""
  email
   成功：null
  password
   失敗："12234"
 名前重複チェック
  引数がusername
  失敗時にExistEx
 メールがあるとき重複チェック
  引数がemail
  失敗時にExistEx
 パスワードハッシュ化
  ハッシュ化されたデータと平文が一致しないこと
 userService呼び出し(create)
  引数がcreateUserInput
  
 jwt発行
  sign関数が呼ばれていること
  
ログイン
 バリデーション
  name
   失敗：""
  email
   成功：null
  password
   失敗："12234"
 compare
  正しいパスワードで認証されること
  誤ったパスワードで失敗すること
 jwt発行（成功した場合のみ）
  sign関数が呼ばれていること
  
  
E2E

正常系
 登録
  正しいデータで入力してdbにuserが作成されること、DBのpwがhash化されていること、jwtが返却されをdecodeしてペイロードにpasswordがないこと（jwt以外ないこと）、201が返ること
  
 ログイン
  正しいデータで入力してjwtが返却され、ペイロードにパスワードが入っていないこと、200が投げられること
  



異常系
 登録
  不正な値で入力するとバリデーションエラーが発生し、400が返ること
  既に存在するusername, emailの時に各AlreadyExistExが投げられること、409が返ること
 ログイン
  不正な値で入力するとバリデーションエラーが発生し、400が返ること
  誤った認証データでログインを失敗すること。jwtが発行されないこと、401が返ること
  
  
Guard系（これは各機能において試験する。）
 tokenあり → 保護API成功
 tokenなし → 401
 不正token → 401
