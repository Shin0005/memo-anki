Deck試験観点
単体試験ではserviceは例外とprisma呼び出し引数だけ。
controllerは一部バリデーションとサービス呼び出し引数だけ

層をまたいだロジックが存在しないので結合は行わず、E2Eを実施する。
E2Eにおいて項目は各CRUDの正常と異常に分かれる。
正常はDb更新されて、レスポンスが正常に返ること。
異常はすべての例外は網羅せず一メソッドに一つの例外くらいが投げられるか確認すること
その他として、認可（他人のdeckを触れない）、nameの重複

E2E

正常系
 Create
  正しいuseridでデータで入力してdbにDeckができて、201が返ること、件数が1件増えていること、返却値とDBの値が一致していること。
 Get
  正しいuseridで入力してそのユーザのすべてのDeckが取得できて、200が返ること
  あるユーザのDeckを取得した際に別UserのDeckが混じっていないこと
 Update
  正しいuseridとデータ(deckid含む)でDeckを更新できて、200が返ること、対象レコードが更新されていること、ほかのレコードに影響がないこと
 Delete
  正しいuseridとdeckidでDeckを削除できて、204が返ること、dbから削除されていること

異常系
 Create
  不正な値で入力するとバリデーションエラーが発生し、400が返ること
  すでに存在するdecknameの場合にAlreadyExistExが発生し、409が返ること
 Get
  ログインしていない状態でguardのエラーが出て、401が返ること（認証で行うので不要!）
 Update
  不正な値で入力するとバリデーションエラーが発生し、400が返ること
  すでに存在するdecknameの場合にAlreadyExistExが発生し、409が返ること
  そのユーザに属さないdeckidを入力してNotFoundExが発生し、404が返ること
 Delete
  不正な値で入力するとバリデーションエラーが発生し、400が返ること(deckid)
  そのユーザに属さないdeckidを入力してNotFoundExが発生し、404が返ること
