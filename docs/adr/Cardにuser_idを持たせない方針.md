## Cardにuser_idを持たせない方針

## 背景

UserとCardとDeckが存在して、Userが複数のDeckを持ち、Deckが複数のCardを持つ。という三者の関係がある。ここでUserは間接的にCardを持つことが分かる。
ここでCardは一つのDeckにしか属せない縛りを作る必要があった（Deck間でCardを共有すると状態の整合性を保つのが難しいため）。
そこで、CardにuserIdを持たせるかを決める必要があった。

## 結論

Cardテーブルにuser_idは保持しない

## 理由

・deckを介してuserが一意に決定でき、冗長となるため

・冗長なカラムにより整合性不一致のリスクが生じるため

・不正な関連付け（異なるuserへの紐付け）を防ぐため

## 他の案

・Cardにuser_idを保持

→ deck.user_idとの不整合リスクがあり、冗長なため不採用
また、cardにuserIdを持たせると、cardを別ユーザのデッキに登録できる恐れ

## 影響

・user単位でcardを取得する際にdeckとのJOINが必要になる
→今のところ必要ない

・データ整合性が構造的に担保される

## 補足

JOINコストは現状許容とし、性能要件が厳しくなった場合のみ非正規化（user_id追加）を検討する
