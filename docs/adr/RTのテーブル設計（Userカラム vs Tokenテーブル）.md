## RefreshTokenの保存方式（Userカラム vs Tokenテーブル）

## 結論

RefreshTokenはUserテーブルのカラムとして保持する

## 理由

・複数デバイスは想定しておらず、複数トークン管理が不要なため

・実装コストを抑え、シンプルに管理するため

## 他の案

・Tokenテーブルを作成しデバイスごとにRTを管理

→ 複数デバイス対応は可能だが、実装および管理コストが増加するため不採用

## 影響

・実装が簡素化される

・1ユーザ1トークンの制約となる

・複数デバイスでの同時ログインに対応できない（やるとrefreshが効かなくなる）

## 補足

RTはCookieに保存されるためデバイス間で共有されず、本構成では整合性を保てないため単一デバイス前提とする
